1. Product Overview
1.1 The Problem
Small and mid-sized staffing agencies — particularly those in cleaning, housekeeping, and security — currently manage their shift operations through WhatsApp groups. A manager drops a message like “Need 3 cleaners Monday 8am at the Hilton” into a group chat with 40+ workers. Multiple workers reply “I’ll take it!” simultaneously, no one knows who actually got the shift, there is no audit trail, and payroll is calculated manually in spreadsheets.
This approach creates five critical problems:
	•	No accountability — workers claim shifts but don’t show up, with no record of commitment
	•	No proof of attendance — agencies have no way to verify workers actually arrived
	•	Payroll chaos — hours are tracked manually, leading to disputes and errors every pay period
	•	No client management — contracts, complaints, and renewals are tracked in the manager’s head
	•	No scalability — an agency with 100+ workers cannot function on WhatsApp
1.2 The Solution
ShiftHub is a multi-tenant SaaS platform that replaces WhatsApp-based shift management with a structured, real-time system. Each agency gets their own isolated workspace where managers create and publish shifts, workers accept them through a mobile app, timesheets are recorded with clock-in/out times, and payroll is calculated automatically from approved timesheets.
The platform also enables workers to hold memberships in multiple agencies. Like Slack, one login gives access to separate workspaces. Workers manage their own schedules across agencies — the system informs but does not enforce. The platform includes a built-in CRM module so agencies can manage their entire client lifecycle from lead generation through contract management to invoicing, all connected to operational data that proves service quality.

2.2 Multi-Tenancy & Multi-Organization Membership
ShiftHub uses a shared database with tenant-scoped query filters. The innovation is that workers are decoupled from organisations through an OrgMembership bridge table.
Before: User.OrgId = Guid (worker belonged to one agency)
After: User has many OrgMemberships (worker can join multiple agencies). Each OrgMembership tracks the worker’s role (Worker, Manager, Admin) in that agency.
When Amina logs in, the app loads her OrgMemberships and opens a workspace selector (like Slack). She chooses SparkleClean or FreshStart. The JWT token is issued with the selected OrgId, and all subsequent queries are scoped to that agency. When she switches workspaces, a new token is issued.
Each agency sees only their own data. Neither knows about the other. Complete isolation, even though both agencies share the same database.
2.3 Authentication Flow
Two authentication paths serve different user types:
	•	Workers (mobile app): Phone number + OTP (one-time password via SMS). After OTP verification, the server loads all the worker’s OrgMemberships. The app shows a workspace picker. Once a workspace is selected, the server issues a JWT token containing the user’s Id, selected OrgId, and Role.
	•	Managers/admins (web dashboard): Email + password with optional 2FA. After login, they see only the workspace(s) they belong to. The token includes OrgId and role.

3. Object Model
3.1 Understanding PK and FK
PK (Primary Key) is a unique identifier for each record in a table. Think of it like a National Insurance number — no two records can ever share the same PK. Every entity gets a Guid (globally unique identifier) as its PK.
FK (Foreign Key) is a reference that points to another table’s PK. When a Shift has SiteId as an FK, it’s saying “I happen at this particular site.”

4. Frontend Design
The platform has three frontends: a mobile app for workers and a web dashboard for managers, plus a personal workspace switcher.
4.1 Workspace Switcher (Mobile & Web)
Mobile app
At the top of every screen, a workspace button shows the current agency’s name and logo. Tapping it opens a switcher that lists all the user’s OrgMemberships. She sees her role in each agency, any notification badges indicating new shifts, and an option to “Join another agency.” Switching workspaces instantly swaps the context — all shifts, pay, and data below are now from the newly selected agency.
Think of it like Slack: your phone has one app, but you can be a member of multiple workspaces. Tapping the workspace picker at the top shows all your communities.
Web dashboard
The web interface also shows the active workspace at the top left with a dropdown to switch. Managers see only the workspace(s) they belong to. If a manager is only part of SparkleClean, they don’t see a switcher — just the agency name and logo.
4.2 Worker Mobile App
Home / Available shifts
Shows a list of available shifts in the currently active workspace, sorted by date. Each shift card displays site name, date, start/end time, pay rate, and slots remaining. Workers tap a shift to see details and accept. Once accepted, the shift moves to the “My shifts” tab.
My shifts / Weekly schedule
A calendar view showing all shifts the worker has accepted in this workspace, sorted by date. Today’s shift is prominently displayed with a “Clock In” button. Workers can review their upcoming schedule and see any conflicts they may have (visually in the calendar). If they realise a clash, they can release one shift to free it up.
Clock-in/out
When a shift is about to start, a “Clock In” button becomes active. The worker taps it and records a timestamp with no GPS requirement. Clock-out works the same way, showing total hours on confirmation.
Pay and payslips
Current period earnings (running total of hours times rate for approved timesheets). Payslip history with download buttons. Each payslip is specific to this workspace — when Amina switches to FreshStart, she sees FreshStart’s payslips. No mixing.
Personal calendar (client-side only)
A private view that aggregates shifts from all the worker’s workspaces into one calendar. This is purely a client-side feature — the app fetches shifts from each workspace separately, merges them locally, and displays them colour-coded by agency. If two shifts overlap in time, the calendar highlights the clash with a gentle nudge: “You have overlapping shifts on Friday. You could release one.”
This is Amina’s personal tool only. No agency can see it. No data is sent to the backend about her cross-workspace schedule. She can release a shift from this view if she spots a conflict.
Profile and settings
Update contact info, upload qualifications, set notification preferences, and mark unavailability. These are all profile-wide (not workspace-specific).
4.3 Manager/Admin Web Dashboard
Dashboard home
An overview of the currently active workspace. Today’s shifts, how many workers are clocked in, any no-shows, key metrics, and upcoming alerts. Quick actions to create a shift or run payroll.
Shift management
Create, publish, and manage all shifts in this workspace. Calendar view showing shifts colour-coded by status. Each shift shows accepted workers, real-time clock-in status, and any no-shows.
Workers / Team management
Worker directory filtered to show only workers in this workspace. Their personal info, qualifications, shift history, and reliability score within this agency. Note: the system does not know or display the worker’s involvement with other agencies.
Timesheet approval
List of pending timesheets for this workspace. Manager reviews and approves. No cross-workspace overlap detection — the system never checks whether a worker has a shift at another agency.
Payroll
Run payroll for the agency on their schedule (weekly, monthly, etc.). All calculations and payslips are specific to this workspace.
CRM, complaints, invoicing, reporting
All CRM features operate within this workspace only. Complaints, contracts, leads, invoices, and analytics are all scoped to the current agency.

5. Backend Breakdown
5.1 Shift lifecycle
The same as before, with one key difference: no overlap detection across agencies.
	•	Manager creates shift — CreateShiftCommand validates the data and creates a Shift with status Draft.
	•	Manager publishes shift — PublishShiftCommand changes status to Published. A domain event triggers notification handlers to send push notifications to all active workers in this workspace.
	•	Worker accepts shift — AcceptShiftCommand checks: (a) are there slots available? (b) does the worker have an assignment for this shift already? (c) is the worker’s membership in this workspace still active? If all pass, ShiftAssignment is created. No cross-agency checks.
	•	Worker clocks in/out — ClockInCommand and ClockOutCommand record timestamps. No GPS verification. Timesheet is created with ClockIn/ClockOut times. HoursWorked is calculated.
	•	Manager approves timesheet — ApproveTimesheetCommand sets IsApproved to true. This timesheet is now eligible for payroll.
	•	Payroll processes — RunPayrollCommand collects all approved timesheets for this workspace and pay period, calculates pay with tax/NI, and generates payslips.
5.2 Multi-tenancy with OrgMembership
Every API endpoint requires authentication. The JWT token contains UserId and the currently active OrgId (selected when the workspace was switched). TenantResolutionMiddleware validates that the user has an active membership in that OrgId before proceeding.
All queries are automatically filtered to the active OrgId via EF Core global query filters. A manager at SparkleClean can never accidentally see GuardForce’s shifts or workers.
5.3 Personal Calendar (Client-Side Only)
This is purely a frontend feature. When Amina opens “My calendar,” the mobile app makes separate API calls to fetch shifts from each of her workspaces. For example:
	•	GET /api/shifts/my-assignments?workspaceId=sparkleclean-org-id (returns Amina’s SparkleClean shifts)
	•	GET /api/shifts/my-assignments?workspaceId=freshstart-org-id (returns Amina’s FreshStart shifts)
The app merges these results locally and renders them in a combined calendar view. No backend logic knows about the other shifts. No data leakage. When Amina spots an overlap and taps “Release” on a shift, the app calls the API for that shift’s workspace only: POST /api/shifts/{id}/release. That workspace updates the assignment status and notifies other workers. Done.
5.4 Releasing Shifts
A new action in the shift lifecycle. ReleaseShiftAssignmentCommand sets the assignment status to Released and reopens the slot in the shift. Other workers get a push notification: “A slot has opened up at [site]”. The worker who released it doesn’t get penalised by the system — it’s her choice.
